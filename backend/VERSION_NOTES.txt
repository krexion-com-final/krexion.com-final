# v2.6.15 — 2026-02 — Duplicate-IP false-positive fix (pre-browser
# probe redundancy elimination)
#
# CUSTOMER-FACING FIX (field report — samsclub01 campaign, Traxun tracker):
#
#    Report:  RUT job with DataImpulse rotating gateway + samsclub01 offer
#             (track.traxun.online). Every visit landed with HTTP 403
#             "Duplicate IP" — 20/20 visits burnt exit IPs, job dead at
#             ~5-6 visits.
#
#    Root cause:  The visit flow made THREE HTTP hits to the offer domain
#                 from the same exit IP within ~2 seconds:
#                    hit #1 — `_probe_proxy_target_reachable` reachability
#                             probe (v2.6.11 fix: domain root only)
#                    hit #2 — `_probe_offer_duplicate_via_proxy` pre-browser
#                             duplicate-check probe (also domain root)
#                    hit #3 — Chromium `page.goto(tracker_url)` real click
#                 Strict affiliate trackers (Traxun, Voluum, RedTrack,
#                 Binom) treat consecutive same-IP hits within a short
#                 window as duplicate sessions, so the browser visit at
#                 hit #3 got flagged.
#
#    Fix:     The pre-browser duplicate probe (hit #2) is redundant with
#             the post-load block detector, which reads the browser's OWN
#             goto response body (no extra proxy request). Disabled the
#             pre-browser httpx dup probe entirely — the browser goto is
#             now the SOLE tracker-touching hit per visit, matching what
#             strict duplicate detectors expect (1 IP == 1 click).
#
#    Effect:  One redundant httpx GET removed per visit → the tracker's
#             duplicate counter no longer trips on setup activity. Post-
#             load block detection is fully unchanged (still catches
#             genuine blocks via browser response body). ProxyJet retry
#             behaviour and rotating-gateway retry behaviour unchanged.

# v2.6.14 — 2026-02 — Rotating-gateway proxy exhaustion fix + offer-block auto-retry for gateway mode
#
# CUSTOMER-FACING FIXES (from field report — RUT job died at 5/20 visits):
#
#   A) ROTATING GATEWAYS (DATAIMPULSE / OXYLABS / BRIGHT DATA / IPROYAL / SOAX / …)
#      NO LONGER "EXHAUST" AFTER FIRST PICK
#      Report: Customer configured DataImpulse Residential US Sticky-2hr
#         (host: gw.dataimpulse.com:10000, username `sessid.{sid}` template),
#         set Concurrency = 10 + Total Clicks = 20. Job ran ~5-6 visits,
#         then every subsequent slot recorded
#         "failed · No more proxies available (no_repeated_proxy = on)".
#      Root cause: `pick_next_proxy()` added the raw gateway URL to
#         `used_proxy_set` after the first fetch and returned None on
#         every call thereafter. This is correct behaviour for a
#         plain `ip:port:user:pass` static list, but WRONG for a
#         rotating gateway — the gateway URL is meant to be re-dialled
#         over and over, returning a NEW exit IP per TCP session.
#      Fix: `_parse_proxy_line` now emits `is_rotating_gateway`
#         (detected via `_detect_rotating_gateway`) for lines whose
#         host matches a well-known provider suffix (dataimpulse,
#         oxylabs, superproxy, brightdata, iproyal, soax, smartproxy,
#         webshare, packetstream, geosurf, netnut, rayobyte,
#         stormproxies, nimbleway, asocks, infatica, proxycheap,
#         920proxy, coronium, pyproxy, ip2world, spaceproxy, abcproxy,
#         hydraproxy, ipburger, shifter, and ~30 others), whose
#         hostname begins with `gw.`, `gate.`, `pr.`, `residential.`,
#         `rotating.`, `rotate.`, `isp.`, `sticky.`, `proxy.`, or
#         `gateway.`, OR whose username contains a session marker
#         (`sessid`, `session-`, `session_`, `sticky-`, `rotating`).
#         Entries flagged rotating-gateway skip the `used_proxy_set`
#         gate — each pick returns the same URL so the gateway
#         re-dials for a new IP. Static lists still get the original
#         single-use behaviour, so nothing regresses for operators
#         loading a plain ip:port:user:pass file.
#      Result: DataImpulse jobs (and every other well-known gateway
#         provider) now run their full concurrency × total budget
#         instead of dying at 5-6 visits.
#
#   B) OFFER-SIDE "DUPLICATE IP" BLOCK NOW AUTO-RETRIES ON A GATEWAY,
#      NOT JUST ON PROXYJET-ON-DEMAND
#      Report: Even when the gateway kept giving IPs, offers/trackers
#         with strict duplicate-IP filters would reject 5/5 attempts
#         and the visit terminated as `skipped_duplicate_ip` — even
#         though re-dialling the gateway would immediately supply a
#         fresh IP.
#      Root cause: The offer-block retry loop (`_OfferBlockRetryNeeded`
#         wrapper in worker()) was gated on `proxyjet_on_demand=True`
#         only, so rotating-gateway users had `max_offer_retries=1`.
#         The pre-browser httpx duplicate probe (~line 8268) and the
#         post-load block handler (~line 9911) were also gated on
#         `proxyjet_on_demand`, so gateway users didn't get the
#         "silent retry with fresh IP, no wasted browser launch"
#         experience ProxyJet users already had.
#      Fix: Introduced `_can_retry_offer_block =
#         bool(proxyjet_on_demand) or _has_rotating_gateway` locally
#         in `run_real_user_traffic()`. All three gating sites now
#         reference the combined flag. DataImpulse operators see the
#         same silent-retry behaviour ProxyJet operators already had,
#         capped at `proxyjet_unique_retry_cap` (default 50) per
#         visit to prevent infinite loops on unrealistically strict
#         filters.
#      Result: Offer-side duplicate/VPN filters no longer terminate
#         visits on rotating-gateway providers — the engine re-dials
#         the gateway silently for a fresh IP up to 50× before giving
#         up on the slot.
#
#   C) BEHAVIOUR NOTES / TROUBLESHOOTING (for the customer)
#      · If the affiliate report still shows Chrome/WeChat/Facebook
#        instead of TikTok after this release: the RUT-generated
#        visits are NOT reaching the offer conversion page (see
#        RUT report.xlsx column `error` — most likely

#  Feature A — "Popup Work" tool (key: P)
# ══════════════════════════════════════════════════════════════════

#   any visible popup / modal / dialog container (role=dialog,
#   [aria-modal="true"], .modal / .popup / .dialog / .overlay /

#   every clickable element (buttons, links, [role=button], onclick,
#   cursor:pointer close-Xs) and returns a checklist grouped by
#   popup_index. The customer ticks which button(s) to click and
#   each tick becomes a normal click step via the existing /click
#   endpoint — so RUT replay runs identical to any manual click.
#
#   Backend: visual_recorder.detect_popup_buttons() + JS bundle
#            _DETECT_POPUP_BUTTONS_JS
#   Route:   GET /api/visual-recorder/{session_id}/detect-popup-buttons
#   Frontend: TOOLS entry `popup_work` + auto-scan on tool select +
#             rose-tinted checklist panel + "Add N popup clicks" btn.

#  Feature B — "Scan" tool (key: S)
# ══════════════════════════════════════════════════════════════════
#   New tool + endpoint that inspects the element under the click
#   coords WITHOUT recording a step or firing the click on the live
#   page. Returns `text`, `selector` (CSS), `xpath_stable`, `xpath_abs`,
#   `tag`, `attrs`, `bbox`. Uses the same _RICH_ELEMENT_CAPTURE_JS
#   the recorder already uses for click/form_fill/dropdown so the
#   locators returned here are BYTE-IDENTICAL to what would be
#   recorded — customers can paste them into RPA Studio, manual
#   click steps, wait_for_selector / wait_for_xpath, or ad-hoc
#   automation JSON with zero translation.
#
#   Backend: visual_recorder.scan_element_at()
#   Route:   POST /api/visual-recorder/{session_id}/scan
#   Frontend: TOOLS entry `scan` + cyan hint panel + scan-result
#             modal with Copy buttons on every field.

# ══════════════════════════════════════════════════════════════════
#   Prior v2.2.7 sent WM_SETICON only to windows owned by Playwright's
#   `_impl_obj._process.pid`, which points at the Node driver — NOT
#   the Chromium browser process that actually owns the taskbar HWND.
#   Result: many customers still saw the Chrome logo in their
#   taskbar because SendMessage never landed.

#     • Refresh the descendant list every 0.8s so windows spawned
#       later (a torn-out tab, DevTools, print-preview) also flip
#       to the Krexion K icon.
#     • Call `SetCurrentProcessExplicitAppUserModelID` BEFORE
#       spawning Chromium so Windows shell registers the incoming
#       child under "Krexion.BrowserProfile.<label>" from the very
#       first frame paint — no brief flash of the Chrome logo
#       during startup.
#     • New public entry-point `apply_krexion_icon_to_pids(pids, …)`
#       supersedes the single-PID helper (legacy call sites still
#       work — the old function now just delegates to the new one).
#
#   Changed files:
#     backend/browser_profile_launcher.py  (multi-PID discovery)
#     backend/krexion_window_icon.py       (new public API + loop)

# v2.5.1 (2026-01-10) -- Fix Native Windows installer build
# -------------------------------------------------------------
# v2.4.6..v2.5.0 Native Windows builds all FAILED at the
# "Build frontend locally (bulletproof fallback)" step with:
#   Unexpected token 'yarn' in expression or statement.
#   The string is missing the terminator: ".
# Root cause: an em-dash character (U+2014) inside the PowerShell
# `throw "..."` string on build-windows-release.yml:358. When the
# temp .ps1 file GitHub Actions writes is read by Windows
# PowerShell 5.1 as cp1252 instead of UTF-8, the em-dash decodes
# to `a<E2><80><94>` -- three characters that INCLUDE a `"` which
# prematurely closes the string literal, making the next tokens
# (`yarn install may be incomplete"`) unparseable.
# Fix: replace `--` (em-dash) with `--` (two ASCII hyphens) in
# that one string. Comments in the same block are unaffected
# because they contain no quote character.
# This is also the reason customer Electron auto-update is stuck
# on 2.4.9 -- native + electron builds share the same trigger
# (backend/VERSION change), so bumping to 2.5.1 will fire ALL
# THREE workflows (VPS deploy + Native Windows + Electron)
# simultaneously and, with the fix, all three should land green.

# v2.5.2 (2026-01-10) -- Fix Native Windows greenlet compile
#                        AND make GH Release upload resilient
# -------------------------------------------------------------
# Two blockers found + fixed after the v2.5.1 build round:
#
# 1) NATIVE WINDOWS -- greenlet Python 3.14 compile failure
#    Symptom: build-installer job failed at "Bundle Playwright
#    Chromium" with C2079/C2027/C2065 errors from cl.exe while
#    trying to build greenlet-3.1.1 from source. Root cause: the
#    build-installer job never called actions/setup-python@v5,
#    so `python` on the self-hosted Windows PC resolved to the
#    system install at C:\Python314. greenlet's C++ code
#    references CPython internals (_PyInterpreterFrame,
#    Py_C_RECURSION_LIMIT) that were removed/renamed in 3.14, so
#    pip's fallback source build hit hard compile errors and the
#    chromium-bundle folder ended up empty (0 MB).
#    Fix: add `Set up Python 3.11` step in build-installer job
#    before "Set up Node 20". pip then picks up the prebuilt
#    greenlet 3.1.1 wheel and playwright installs without any
#    compilation on the runner.
#
# 2) ELECTRON + NATIVE -- GH Release upload ECONNRESET
#    Symptom: softprops/action-gh-release@v2 succeeds at
#    uploading the small assets (latest.yml, blockmap) then
#    ECONNRESETs mid-transfer on the ~600 MB installer .exe.
#    The release is left as a DRAFT with a partial asset set and
#    customer Electron auto-update polling latest.yml either
#    404s on the missing .exe or never sees the new version.
#    Root cause: the action has no per-asset retry semantics --
#    a single TCP drop from the residential ISP self-hosted
#    runner corrupts the whole release. Wrapping in
#    `nick-fields/retry` around a `uses:` action isn't
#    supported.
#    Fix: mark the softprops step continue-on-error and add an
#    idempotent "Verify + Repair GH Release upload" step
#    wrapped in nick-fields/retry@v3 (3 attempts, 60s wait).
#    The step compares each expected file's local size against
#    the attached asset; if missing or size-mismatched it
#    deletes the stale asset and re-uploads via System.Net.Http
#    HttpClient with a 20-25 min timeout. Once every expected
#    file is attached with the correct size, it PATCHes the
#    release to draft=false. Applied to BOTH build-electron-
#    desktop.yml and build-windows-release.yml.
