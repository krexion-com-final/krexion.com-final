# v2.1.78 — 2026-07-02 — IPv6 support + VPS load-reduction + security hardening
#   Customer reports addressed:
#     A) "Clicks page pe kaafi rows pe IP show ni hota" — IPv6-only
#        mobile-carrier visitors (Jio / T-Mobile / Zong / Airtel etc.
#        via Cloudflare CF-Connecting-IP as an IPv6) were being dropped
#        by the strictly-IPv4 header parser. Every real visitor now has
#        a stored IP.
#     B) "VPS stuck ho jata hai jabki heavy jobs PC pe hain" — the
#        bridge cache pre-warmer was firing one 55-second bridge task
#        per online-heartbeat user every 25 s, saturating MongoDB with
#        polls. Now capped + UI-active-only.
#
#   1) IPv6 SUPPORT (backend/server.py)
#      • get_all_client_ips() rewritten — extracts both IPv4 and IPv6
#        from CF-Connecting-IP / X-Forwarded-For / X-Real-IP /
#        True-Client-IP / peer, normalises v6, keeps the infra-prefix
#        skiplist for false-dup prevention.
#      • click_doc now stores `ipv6` (was indexed but never written).
#      • primary_ip_for_storage falls back to IPv6 when no IPv4 —
#        Clicks table never shows `-` for a real visitor anymore.
#      • Duplicate detection is now IP-family-agnostic: same IPv6
#        repeat visits are blocked (previously allowed because IPv6
#        was explicitly excluded).
#      • ClickResponse Pydantic model exposes the new ipv6 field so
#        the frontend actually receives it.
#      • Block-page display + cookie-fallback IP now surface IPv6 so
#        users no longer see "IP: unknown" on IPv6-only blocks.
#      • Old legacy get_client_ip() reduced to a thin shim over
#        get_all_client_ips().
#
#   2) FRONTEND (ClicksPage.js)
#      • "IPv4" column shows ONLY IPv4 (was falling back to `ip_address`
#        which after the IPv6 fix contained IPv6 — confusing UX). IPv6
#        column continues to render v6.
#      • CSV export toast shows a `truncated` warning when the user
#        has more clicks than the export cap.
#
#   3) VPS LOAD REDUCTION (backend/server.py)
#      • Bridge cache pre-warmer: cycle 25 s → 30 s, max 10 concurrent
#        pre-warm tasks (env: BRIDGE_PREWARM_MAX_PARALLEL), and only
#        fires for users whose FRONTEND is actively polling
#        /api/bridge/me/local-status within the last 3 min (env:
#        BRIDGE_PREWARM_UI_WINDOW_SEC=180). Customers who leave PC on
#        but don't use the cloud UI no longer cost VPS pre-warm cycles.
#      • /api/bridge/me/local-status now has a 5 s in-memory cache per
#        user_id — cuts ~30× MongoDB reads while UI still feels live.
#      • /api/clicks/export switched from `to_list(1_000_000)` to
#        cursor iteration with a hard cap (EXPORT_CLICKS_MAX=100000);
#        response includes `total_available` + `truncated` so the UI
#        can prompt users to narrow the date range. Peak RAM per export
#        is now ≈ 50 MB instead of ≈ 500 MB. This was the biggest single
#        contributor to the "VPS stuck" reports.
#      • /api/clicks default-all path capped at 10 000 rows (was
#        100 000) — UI paginates at 50/page so this is invisible.
#
#   4) SECURITY / STABILITY
#      • Admin password reset now actually works. reset_password wrote
#        the new hash to `admin_config` but admin_login never read it —
#        login always compared plaintext against env ADMIN_PASSWORD, so
#        "Forgot password" was a silent no-op. Login now consults
#        admin_config first, falls back to env only if no override.
#      • Brute-force protection on /admin/login, /auth/login, and
#        /sub-users/login — 8 failed attempts per (email:IP) in a
#        15 min window → HTTP 429 + Retry-After header. Successful
#        login clears the counter. Env-tunable
#        (MAX_LOGIN_FAILURES, LOGIN_LOCKOUT_WINDOW_SEC).
#      • Admin login uses hmac.compare_digest against the env password
#        (constant-time) so timing side-channels are eliminated on
#        the fallback path.
#      • Removed noisy "password match: True/False" log line from
#        admin_login (info leak in supervisor logs).
#      • JWT_SECRET_KEY default `your-secret-key-change-in-production`
#        detected → a random 48-byte secret is generated for that run
#        and a big WARNING is logged. Prevents forged admin JWTs when
#        .env is misconfigured.
#      • ADMIN_PASSWORD == "admin123" default → WARNING logged.
#      • get_country_from_ip() bug fix — the returned `is_proxy` field
#        was mistakenly set to the `is_vpn` value, so real proxy IPs
#        were mislabelled in analytics. One-line fix.
#      • CORS: legacy allow_credentials=True + origins=["*"] was invalid
#        per spec (browsers refuse cookies to wildcard). Now: wildcard
#        implies credentials=False; concrete list implies credentials=
#        True. Set CORS_ORIGINS to your prod origins to enable creds.
#      • Sub-users now receive WebSocket live-click updates. The WS
#        handler used to only look in `users` collection, so sub-user
#        connections closed with 4002. Now honours the is_sub_user JWT
#        claim and joins the parent's channel.
#      • /api/system/install-update `is_admin` check moved BEFORE the
#        cloud-bridge relay — previously a non-admin sub-user on
#        krexion.com could trigger a container rebuild on their
#        parent's PC.
#      • Removed a dead POST /api/admin/releases stub that always
#        raised 500.
#

#     • IPv4 first visit → 302, IPv6 first visit → 302.

# v2.1.69 — 2026-06-28 — Native install fixes (memory + browser profile launch)
#   1. RUT memory throttle no longer falsely fires on Windows native
#      installs. `_detect_container_mem_limit_mb()` previously fell back
#      to a hardcoded 6 GB cap when cgroup files weren't present (which
#      is ALWAYS the case on Windows), causing the customer's 32 GB
#      machine to throttle at backend RSS > 4.9 GB — well within
#      normal Playwright + Chromium territory. As a result every RUT
#      job got stuck on "running · 0/N visits" even though the host
#      had tons of free RAM.
#      Fix: third detection step uses psutil.virtual_memory() when
#      cgroup is absent, deriving cap = (total_ram - 4 GB) × 0.85
#      clamped to [1 GB, 32 GB]. 8 GB customer machines correctly get
#      ~3.5 GB cap (throttles before OOM); 32 GB machines get ~24 GB
#      cap (no false throttle). Cloud VPS path (cgroup) unchanged.
#      Tests: 9/9 in tests/test_native_mem_limit_fallback.py.
#
#   2. Browser Profile launch on NSSM-installed Windows build no longer
#      stuck on "launching..." forever. Root cause: Windows Services
#      run in Session 0 and cannot display GUI windows on the user's
#      desktop. Customer clicked Launch → backend (Session 0) spawned
#      Chromium → process existed but window invisible. Electron build
#      never hit this because its backend is a child of the Electron
#      user-session main process.
#      Fix: NSSM-build backends (KREXION_BUILD_TYPE=binary) now write
#      pending launches into a new `browser_launch_queue` Mongo
#      collection. The tray-app dashboard (already running in user
#      session via HKCU Run autostart) polls every 2s and spawns
#      Chromium IN ITS OWN PROCESS — Session 1, full desktop access,
#      Chromium window appears normally. Stop requests propagate the
#      same way via `stop_requested` flag on the queue record.
#      Tests: 11/11 in tests/test_browser_profile_user_session_handoff.py.
#      Backward compat: Electron + cloud VPS + sync_client bridge all
#      unchanged (defer only fires when both `sys.platform == 'win32'`
#      AND `KREXION_BUILD_TYPE == 'binary'`).

# v2.1.68 — 2026-06-27 — RUT reliability + Live Visual Grid bugfix
#   1. Live Visual Grid no longer hides actively-running visits.
#      `isVisitActive()` was checking per-step `latest_event.status === 'ok'`
#      which fired on EVERY successful automation step — so the grid stayed
#      empty even when 118-step flows were running in the background.
#      Customers saw "Waiting for first visit to start…" forever.
#      Fix: use VISIT-level `v.status` only as the done-signal; the
#      `stage:'done'` short-circuit catches the race where push_live_step
#      emits the final event before the dispatcher flips v.status.
#   2. Header tile counter (✓ / ✗ / ⏵) now uses the same correct signals
#      instead of per-step status. Counters and grid agree again.
#   3. Stuck-watchdog default raised 240s → 600s. Survey-heavy multi-deal
#      offers (like "target 750 v10") routinely have 5-8 min idle pauses
#      on the post-survey wait page — 240s was too aggressive and was
#      mass-aborting otherwise-healthy visits. UI input range also
#      widened to 30-1800s to match the backend cap.
#   4. Mid-flow chrome-error://chromewebdata/ is now tagged as a transient
#      tunnel failure (proxy dropped during form submit). The
#      `target_mode=conversions` dispatcher excludes these from the
#      max_attempts budget so a single proxy hiccup mid-flow no longer
