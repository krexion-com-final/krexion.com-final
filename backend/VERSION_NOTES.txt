# v2.1.69 — 2026-06-28 — Native install fixes (memory + browser profile launch)
#   1. RUT memory throttle no longer falsely fires on Windows native
#      installs. `_detect_container_mem_limit_mb()` previously fell back
#      to a hardcoded 6 GB cap when cgroup files weren't present (which
#      is ALWAYS the case on Windows), causing the customer's 32 GB
#      machine to throttle at backend RSS > 4.9 GB — well within
#      normal Playwright + Chromium territory. As a result every RUT
#      job got stuck on "running · 0/N visits" even though the host
#      had tons of free RAM.
#      Fix: third detection step uses psutil.virtual_memory() when
#      cgroup is absent, deriving cap = (total_ram - 4 GB) × 0.85
#      clamped to [1 GB, 32 GB]. 8 GB customer machines correctly get
#      ~3.5 GB cap (throttles before OOM); 32 GB machines get ~24 GB
#      cap (no false throttle). Cloud VPS path (cgroup) unchanged.
#      Tests: 9/9 in tests/test_native_mem_limit_fallback.py.
#
#   2. Browser Profile launch on NSSM-installed Windows build no longer
#      stuck on "launching..." forever. Root cause: Windows Services
#      run in Session 0 and cannot display GUI windows on the user's
#      desktop. Customer clicked Launch → backend (Session 0) spawned
#      Chromium → process existed but window invisible. Electron build
#      never hit this because its backend is a child of the Electron
#      user-session main process.
#      Fix: NSSM-build backends (KREXION_BUILD_TYPE=binary) now write
#      pending launches into a new `browser_launch_queue` Mongo
#      collection. The tray-app dashboard (already running in user
#      session via HKCU Run autostart) polls every 2s and spawns
#      Chromium IN ITS OWN PROCESS — Session 1, full desktop access,
#      Chromium window appears normally. Stop requests propagate the
#      same way via `stop_requested` flag on the queue record.
#      Tests: 11/11 in tests/test_browser_profile_user_session_handoff.py.
#      Backward compat: Electron + cloud VPS + sync_client bridge all
#      unchanged (defer only fires when both `sys.platform == 'win32'`
#      AND `KREXION_BUILD_TYPE == 'binary'`).
#
# v2.1.68 — 2026-06-27 — RUT reliability + Live Visual Grid bugfix
#   1. Live Visual Grid no longer hides actively-running visits.
#      `isVisitActive()` was checking per-step `latest_event.status === 'ok'`
#      which fired on EVERY successful automation step — so the grid stayed
#      empty even when 118-step flows were running in the background.
#      Customers saw "Waiting for first visit to start…" forever.
#      Fix: use VISIT-level `v.status` only as the done-signal; the
#      `stage:'done'` short-circuit catches the race where push_live_step
#      emits the final event before the dispatcher flips v.status.
#   2. Header tile counter (✓ / ✗ / ⏵) now uses the same correct signals
#      instead of per-step status. Counters and grid agree again.
#   3. Stuck-watchdog default raised 240s → 600s. Survey-heavy multi-deal
#      offers (like "target 750 v10") routinely have 5-8 min idle pauses
#      on the post-survey wait page — 240s was too aggressive and was
#      mass-aborting otherwise-healthy visits. UI input range also
#      widened to 30-1800s to match the backend cap.
#   4. Mid-flow chrome-error://chromewebdata/ is now tagged as a transient
#      tunnel failure (proxy dropped during form submit). The
#      `target_mode=conversions` dispatcher excludes these from the
#      max_attempts budget so a single proxy hiccup mid-flow no longer
